# Roadmap

本文档记录 LLM Project 当前阶段的开发计划。  
项目仍处于 Demo 阶段，规划内容可能根据实际体验、用户反馈和技术实现难度调整。

---

## 版本规则

### Demo 阶段

当前项目处于 Demo 阶段。

```text
Demo 1.x.x
```

用于构建正式版 `v1.0.0` 所需的基础功能。

### 正式版阶段

正式版发布后采用：

```text
vX.Y.Z
```

含义：

- `X.0.0`：某一大方向完成
- `X.Y.0`：通往下一个大方向的重要工程
- `X.Y.Z`：该工程内的优化、修复和细分版本

## Demo 1.2.7：生态引流形象版（当前主攻）

优先级：P0（最高，先于 1.3.x）

### 立项背景

demo1.2.5 / 1.2.6 在社区发布后，浏览量增长但下载与互动极低。
据反馈判断：路人最强的尝试动机是「能不能用上酒馆（SillyTavern）生态的角色卡」，
其次是「第一眼界面是否值得停留」。

因此 1.2.7 的目标**不是冲下载量，而是冲互动率**：
让路人愿意尝试、愿意反馈问题、愿意提建议——
为后续「项目正式命名 / 投票」积累真实关注度。

> 入场券 = 转译（让人愿意进门）；
> 护城河 = 状态栏 + 副本机制 + 全新界面（让人进门后不想走）。

### 对外三钩子（路人能一句话看懂的卖点）

1. **导入第三方角色卡**：整合 PC 转译核心，移动端可直接导入酒馆卡（设定 / 世界书 / 开场白）。
2. **全新界面（UI 大改）**：决定第一眼留存的视觉差异。
3. **状态栏基础版**：让角色状态（好感度 / 心情 / 时间 / 地点等）可见、可注入 Prompt。

### 对内地基（不对外当卖点，但必须先做）

- **会话副本 Prompt 机制（核心地基）**
    - 原卡 system prompt 为「母版」，永不被运行时改写。
    - 进入会话时从母版克隆出「会话副本（active prompt）」，实际注入用副本。
    - 副本可被三股力量改写：① UI 交互 ② 状态栏数值 ③（后续）流动式自动更新。
    - 清空聊天记录时：删除副本 → 重新从母版克隆干净副本（实现「改写后可还原」）。
    - 扩展现有 `_renderPromptTemplate` 宏替换，使其识别来自副本的动态变量。
- 说明：状态栏注入、UI 交互影响 Prompt、未来流动式更新，**全部依赖这块地基**，
  故必须在状态栏之前落地，避免后续版本推翻重做。

### 积木 UI（本版只做样板，不做平台）

- 本版**不做**通用积木编辑器 / 资产库 / 转译自动生成。
- 仅做一个**写死的可玩样板**，跑通完整闭环：
  「开场白 UI（如输入主角姓名 + 选择技能 + 换一批 + 确认）→ 写入会话副本 → 真正影响 Prompt → 清空记录后还原」。
- 目的：可截图、可录屏、引发「以后能不能自己拼？」的讨论 = 制造互动诱饵，同时验证副本地基。

### 发布后动作（独立进行，不绑发版）

- 1.2.7 推文尽量详尽地写特色与好玩的点。
- 发布后另发一条**专门的「项目起名 + 投票」推文**，
  用项目最缺的「名字」换取最想要的「互动」。

### 风险控制

- 转译已基本完成（低风险）；状态栏中等；UI 大改耗时不可控，优先保发版日期。
- 积木严格限定为样板，**严禁在本版承诺平台化**，防止再次「规划过于理想」导致拖期。

## Demo 1.3.1.5：第三方角色卡迁移工具（005）

优先级：P1（已提前，因生态导入是当前最强下载动机）

定位：

```text
教程 = 让用户会用；迁移工具 = 让用户有内容可用。
```

先做 PC 工具，把兼容逻辑调稳，再裁剪整合进移动端。
不执行酒馆脚本 / 正则 / 扩展插件，只做解析 + 规则映射 + 保留。

### 005-A：规则转换核心（已完成）

- 纯 Dart 转换核心（PC / 移动端共用）
- PNG 内嵌 `chara` / `ccv3` 读取
- SillyTavern V2 / TavernAI V1 JSON 解析
- 字段映射到 LLM Project 角色 + 内嵌世界书
- 输出 `.llmcard` + 转换报告
- 桌面拖拽 / 文件夹批量、移动端多选
- 单元测试

### 005-B：AI 智能归类（规划）

- 用户自配 API；把混在一个字段里的多类设定拆分归类
- 允许最小必要改写（补主语 / 换代词 / 调语序），禁止新增事实
- 每段带 `source_refs` + `transform_type` + `risk`

### 005-C：AI 检查精修 + 源文对照（规划）

- 覆盖检查（漏转）/ 忠实性检查 / 新增事实检查 / 重复归类检查
- 左原始字段 / 右转换结果对照视图

### 005-D：移动端整合（规划）

- 「导入角色卡」自动识别第三方格式并复用同一套核心

详见 `docs/CHARACTER_CONVERTER.md`。

## Demo 1.3.2：流动式 Prompt 更新版

优先级：P2

> 前置：1.2.7 已落地「会话副本 Prompt 机制」与「状态栏基础版」。
> 本版在副本地基上加入「自动流动更新」。

计划内容：

- 固定回合数后，LLM 判断会话内容，生成对会话副本的更新
- 更新写入副本，**不覆盖母版**（清空记录仍可还原）
- 副本更新与状态栏数值、UI 交互变量的合并 / 去重规则
- 更新前后差异可见（可选：用户确认后再应用）
- 完善状态栏（关系阶段 / 状态备注 / 状态编辑面板等增强项）

目标：

让系统 Prompt 随剧情自然演化，又不污染原始角色卡。

## 声明式 UI 积木系统（平台级，多版本推进）

定位：从 1.2.7 的「写死样板」逐步演化为可拼装的可视化平台。
明确这是**跨多个版本的工程**，不在任何单一版本内承诺完成，避免规划理想化。

分层架构（关键设计）：

- **第 1 层：原子积木（原语）**
    - 进度条 / 标签行 / 按钮组 / 文本块 / 文本输入 / 图片槽 / 折叠面板 / 状态点 等。
    - 用 Dart 实现，随 App 版本发布逐步累积（真正能「越做越多」）。
    - 起步克制：先做能覆盖 ~80% 卡的 6~8 个原语。
- **第 2 层：复合模块 / 模板（资产）**
    - 用第 1 层原语组合 + 字段绑定 + 样式参数，**纯数据（JSON）描述**。
    - 可由 AI 转译生成、用户导入即入库、可热更、可分享。
    - **不下发可执行代码**（移动端 AOT 无法运行时编译 Dart；且第三方代码 = 安全风险）。
- **引擎**：App 内置「积木解释引擎」，读第 2 层数据 → 调第 1 层原语渲染（思路同官方 RFW）。

交互影响 Prompt：

- 积木交互（输入 / 选择等）→ 写入「会话副本 Prompt」→ 经宏替换注入 → 清空记录后还原。
- 依赖 1.2.7 的会话副本地基。

转译映射（最难、靠后）：

- 第三方卡的 `regex_scripts` + 世界书「输出格式」描述的是**数据契约**（有哪些字段 / 选项 / 交互），
  CSS/HTML 只是**皮肤**。
- 转译做的是：读懂数据契约 → 丢弃皮肤 → 查资产库决定「复用已有模板 or 用原语新建模板」。
- 能力边界（已确认，不动摇）：
    - 🟢 结构化数据面板（状态栏 / 属性 / 选项 / 档案）：可达 ~90%。
    - 🟡 仿真界面（聊天气泡 / 立绘切换 / 仿社交 App）：可达 ~60-75%，精细动画 / 像素级皮肤做不到。
    - 🔴 JS 驱动游戏逻辑（MVU / tavern_helper / 平台级自研引擎）：不转，仅识别 + 报告。

资产治理（必须早埋）：

- 跨版本降级：缺少新原语时退化为文本 / 占位，不崩。
- 资产去重：语义标签 + 相似度匹配，支撑「复用还是新建」的判断质量。

## 长期方向

正式版后可能继续推进：

- 知识库
- 破限库
- Prompt 路由器
- 智能记忆系统
- 模糊回忆
- 背包 / 数值结算等独立玩法系统（与积木 UI 解耦，单独立项）
- 更丰富的角色状态与剧情状态管理

---

## 当前原则

- 先保证本地工具稳定
- 不托管用户内容
- 不内置官方社区
- 资产分享采用文件流通
- 导入时严格检查内部标识
- 尽量兼容旧版本资产格式
